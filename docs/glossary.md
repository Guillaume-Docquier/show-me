# Glossary

These are terms and their meaning in the context of this app.

| Term               | Description                                                                                                                                                                                                                                 |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Analysis           | The versioned, language-neutral description of project files, metrics, dependencies, coverage, and diagnostics. It is the only project payload embedded in a report.                                                                        |
| Consumer           | A project file with a runtime dependency on another project file.                                                                                                                                                                           |
| Coverage           | The percentage of executable lines covered according to a selected Istanbul or LCOV report. A project file absent from coverage data has unknown coverage, not zero coverage.                                                               |
| Dependency         | A directed relationship from a source project file to another project file or external package that the source uses at runtime. An ESM import is one language-specific example.                                                             |
| Discovery          | Filesystem traversal that identifies candidate project files using language support, ignore rules, and standard exclusions without parsing source syntax.                                                                                   |
| External package   | A canonical third-party package root referenced by a project file, such as `react` or `@scope/package`. Subpaths collapse to one fixed-size synthetic node; installed files are never analyzed.                                             |
| File selection     | A typed analysis policy applied after permanent discovery exclusions. It currently controls whether conventionally named test files are included; CLI ignore patterns and persistent configuration remain planned.                          |
| Graph node         | A visual entity in the report. A node is explicitly either a project file or an external package and has a collision-proof type-prefixed presentation identity.                                                                             |
| Language module    | An internal project-level analyzer that understands the source files and dependency rules of one language family.                                                                                                                           |
| Layout             | The deterministic placement of graph nodes in two dimensions, including collision spacing derived from their visual sizes.                                                                                                                  |
| Line category      | Exactly one classification assigned to each physical line: code, comment, or blank. Code takes precedence when a line also contains a comment.                                                                                              |
| Line metric        | A code, comment, or blank physical-line count attached to a project file and available for display or node sizing.                                                                                                                          |
| Node size          | A renderer-neutral radius-like value that grows logarithmically with the active line metric so ordinary files remain visually comparable while very large files stand out.                                                                  |
| Presentation model | Browser-owned graph data derived on report load from analysis, including graph identities, visual sizes, colors, display paths, relationship indexes, and interaction metrics. It is not embedded or produced by Node.                      |
| Physical line      | Source text between LF, CRLF, or CR separators. Empty input has no physical lines, and a final separator does not create a trailing phantom line.                                                                                           |
| Project file       | A source file owned by the analyzed project, supported by a language module, and not excluded by discovery rules. Executable JavaScript and TypeScript files are the initial examples.                                                      |
| Project root       | The directory selected for analysis. It defaults to the directory where `show-me` is invoked.                                                                                                                                               |
| Report             | The self-contained HTML file that safely embeds analysis, styles, and the browser bundle and can be opened independently without project source files or a server.                                                                          |
| Runtime dependency | A dependency that can affect runtime behavior. A static non-type-only ESM import is one language-specific example.                                                                                                                          |
| Test file          | A supported JavaScript or TypeScript file whose basename contains `.test.` or `.spec.`, case-insensitively and anywhere in the basename. It is excluded by default; directory names and bare `test.ts` or `spec.ts` basenames do not match. |
| Workspace package  | A package declared by a pnpm workspace. A later milestone assigns project files to packages so the visualization can filter by workspace package.                                                                                           |
